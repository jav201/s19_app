> - `shall` = normative, binding, verifiable requirement. **Only** inside HLR / LLR statements.

## 1. Introduction

### 1.1 Purpose
This document captures the Phase-1 requirements for batch 2026-06-11-batch-08 of `s19tool`: wiring **placeholder operation classes** for special operations on loaded HEX/S19 images (CRC, extract, split-per-memory-segment), so that each operation can later receive real requirements without re-plumbing the app. It is the input to the Phase-2 independent review and the Phase-3 implementation.

### 1.2 Scope
**In scope (this batch):**
- A NEW `s19_app/tui/operations/` package defining a minimal operation abstraction (`Operation` interface), a structured result envelope (`OperationResult`), and three placeholder operation classes (`crc`, `extract`, `split_by_segment`) whose `execute` is an identity passthrough over the loaded image snapshot.
- A deterministic, code-driven operation registry (no model-driven routing).
- A headless service-level entry point (`run_operation`) following the existing `tui/services/` seam pattern.
- **A dedicated TUI operations view** (gate decision 2026-06-11 — G-1 REVERSED, §6.3): select one registered operation, execute it through `run_operation`, and view the result (`status`/`notes`) plus the resulting image rendered with the existing hex viewer (`render_hex_view_text`, `s19_app/tui/hexview.py:324`). HLR-004.
- Unit tests in a NEW `tests/test_operations.py`; pilot (headless `run_test`) tests in a NEW `tests/test_tui_operations_view.py`.

**Out of scope (explicitly NOT this batch):**
- Any real operation logic (CRC algorithms, extraction, segment splitting). Placeholders return the input unchanged — the operator's gate rationale, recorded verbatim in §6.2 C-4, is that the operations are real future work needing proper definition before implementation; this batch builds only the control/log plumbing around them.
- Any CLI subcommand for operations (the CLI half of former G-1 stays deferred).
- Result export, operation history, per-operation configuration/parameters, or progress UI — the placeholder result shown over the unchanged image IS the acceptance demo.
- Any file emission or re-serialization of the image (S19 or HEX) by an operation (design decision C-3, §6.2). Note: an Intel HEX emitter does not exist in the repo (probe P3, §6.2 C-3).
- US-006 (hex compare mode) — registered as a queued story, operator decision at batch-08 init.

## 2. Overall description

### 2.1 Product perspective
`s19tool` has three layers: parsers (`S19File`, `s19_app/core.py:218`; `IntelHexFile`, `s19_app/hexfile.py:20`) → range/validation engine → TUI services + view code. Batch-07 added the `s19_app/tui/changes/` package (change documents, apply engine, check engine) and consolidated the rule that feature logic lives in `tui/services/` modules that the ~5k-line `S19TuiApp` merely orchestrates. This batch adds a fourth, deliberately thin, capability family — *operations over a loaded image* — as a sibling package `s19_app/tui/operations/` (NEW) plus one service module, mirroring the proven `changes/` + `change_service` shape. No existing module is modified; the batch is purely additive (one consciously accepted exception may arise in Phase 3 if `tui/services/__init__.py:1` needs an export line — it is currently a 1-line file).

### 2.2 Product functions
- F1 — Define the operation abstraction: a named operation with a human-readable description and an `execute(loaded) -> OperationResult` contract.
- F2 — Provide three placeholder operations (`crc`, `extract`, `split_by_segment`) wired end-to-end, each an identity passthrough.
- F3 — Provide a deterministic code-driven registry to enumerate and look up operations.
- F4 — Provide a headless service entry point so a follow-up batch (or a test) can run an operation without any TUI interaction.
- F5 — Provide a dedicated TUI operations view: list the registered operations, execute the selected one through the F4 service, and present the `OperationResult` (`status`/`notes`) together with the resulting image rendered by the existing hex viewer. (Added at the Phase-1 gate — G-1 reversal, §6.3.)

### 2.3 User characteristics
Single operator (s19tool user, embedded-firmware context). Technically proficient; consumes this batch directly through the F5 operations view (select → execute → see the unchanged image plus `status="placeholder"`), which proves the pipeline end-to-end. The deeper deliverable remains plumbing whose visible effect is that the NEXT batch can fill one placeholder with real requirements touching exactly one class body.

### 2.4 Constraints
- **Stack:** Python 3.11+, pytest (measured 9.0.3, 2026-06-11 via `python -m pytest --version`; unpinned in project files) (ratified runner — `pyproject.toml` line 40, `[tool.pytest.ini_options]`; CI gate: PRs run `pytest -q -m "not slow"`). No new dependencies. (Informative: pytest itself is unpinned in project files — pre-existing condition, unchanged this batch.)

- **Size:** small batch — 4 HLRs / 13 LLRs (HLR-004 + 4 LLRs added at the gate); placeholders must not speculate beyond the story (no result fields that only a specific real operation would need), and the view must not speculate beyond select/execute/show (no export, history, or configuration — §1.2 out-of-scope).

### 2.5 Assumptions and dependencies
- A-1: `LoadedFile` (`s19_app/tui/models.py:9`) remains the canonical loaded-image snapshot, produced by `build_loaded_s19` (`s19_app/tui/services/load_service.py:17`) and `build_loaded_hex` (`load_service.py:44`). - A-2: Exactly one placeholder receives real requirements in a follow-up batch (operator statement, §2.6). The abstraction is sized for that, not for a plugin ecosystem.
- A-3: The operator accepts in-memory identity semantics (not byte-identical file re-emission) as the placeholder contract — recommended in C-3 but **gate-confirmable** (G-2, §6.3).

> Each HLR is an EARS statement. Allowed patterns:
>
> - **Ubiquitous:** `The <system> shall <response>.`
> - **Event-driven:** `When <trigger>, the <system> shall <response>.`
> - **State-driven:** `While <state>, the <system> shall <response>.`
> - **Optional feature:** `Where <feature is included>, the <system> shall <response>.`
> - **Unwanted behavior:** `If <unwanted condition>, then the <system> shall <response>.`
> - **Complex:** combinations of the above.

> **Testing-strategy note** — every `test (...)` label below targets pytest (measured 9.0.3, 2026-06-11; unpinned in project files), the strategy-ratified runner (`pyproject.toml` `[tool.pytest.ini_options]`, line 40; CI workflow `.github/workflows/tui-ci.yml` runs `pytest -q`). ### HLR-001 — Operation abstraction with identity-passthrough placeholder contract (US-007)
- **Traceability:** US-007; §6.2 C-1 (chosen shape), C-2 (result contract), C-3 (identity semantics).
- **Statements (EARS):** (1) The system shall define an operation abstraction under which every operation exposes a stable string `operation_id`, a stable human-readable `title`, a human-readable description, and an `execute` method with signature `execute(loaded: LoadedFile, *, now_fn: Optional[Callable[[], datetime]] = None) -> OperationResult` — taking one loaded-image snapshot (`LoadedFile`) plus a keyword-only injectable clock defaulting to the system UTC clock (the LLR-001.2 `now_fn` seam), and returning a structured `OperationResult`. (2) When a placeholder operation's `execute` is invoked with a loaded-image snapshot, the operation shall return an `OperationResult` whose `output` is the same snapshot object and shall leave the snapshot's `mem_map`, `ranges`, and `errors` unmutated. (3) The `OperationResult` shall carry the canonical field set of §6.2 C-2 (`operation_id`, `status`, `input_path`, `variant_id`, `output`, `notes`, `timestamp_utc`) with `status` drawn from the closed domain {`placeholder`, `ok`, `error`} and a deterministic `to_dict()` serialization.
- **Rationale (informative):** the operator's working example is "identity passthrough"; the envelope (rather than bare passthrough functions) is what makes the later fill-in a one-class-body change — see §6.2 C-1 for the alternatives comparison the operator requested.
- **Validation:** test · **Executed verification:** `pytest -q tests/test_operations.py` (NEW — created in Phase 3; probe P5 run 2026-06-11: `python -m pytest -q tests/test_operations.py` reports `no tests ran in 0.00s` pre-implementation) · **Numeric pass threshold:** 100% pass of TC-001/TC-002/TC-003/TC-009 (0 failures, 0 skips). · **Priority:** Must.

## 4. Low-level requirements (LLR)

> Each LLR decomposes an HLR into a verifiable property at the implementation level. Same EARS regime. ID format: `LLR-<HLR>.<M>`.

### HLR-001 LLRs — abstraction, result envelope, identity

- **LLR-001.1 — Operation interface.** The system shall define an `Operation` abstract base class (NEW — created in Phase 3, in `s19_app/tui/operations/model.py`, NEW) whose concrete subclasses provide: a class-level `operation_id: str`, a class-level `title: str`, a `describe() -> str` method returning a non-empty human-readable description, and an `execute(loaded: LoadedFile, *, now_fn: Optional[Callable[[], datetime]] = None) -> OperationResult` method (the keyword-only `now_fn` is the LLR-001.2 clock seam; callers omitting it get the system UTC clock), where `LoadedFile` is the existing snapshot dataclass (`s19_app/tui/models.py:9`).

### 6.1 Extended glossary
See §1.3. Additionally: **"fill-in batch"** — the future batch that replaces exactly one placeholder's `execute` body (and possibly its result `notes`/`status`) with real behavior; the design intent of this whole batch is that the fill-in touches `placeholders.py` (one class) plus its tests, and nothing else.

### 6.2 Relevant design decisions

**C-1 — Operation ABC + result envelope + code-driven registry (chosen over literal passthrough functions).** The operator's working example was option (a): three literal identity functions `f(loaded) -> loaded`. Evaluated alternatives:
  - *(a) Literal identity functions.* Cheapest today, but it encodes the wrong signature: a real CRC does not return the input image — it returns a computed value plus metadata; a real split returns multiple artifacts. Filling in any placeholder would change the function signature and therefore every caller — exactly the re-plumbing the story exists to avoid. Rejected: fails the benefit clause.
  - *(b) `Operation` ABC + `OperationResult` envelope + deterministic registry.* The envelope absorbs future variation (`status`, `notes`, and — when a real op needs it — additional payload via a widening of the C-2 contract under the contract-touch rule) so the fill-in changes one class body. Registry keeps dispatch greppable and code-driven (engineering rule 5: no model routing).
  - *(c) Codebase-idiom check.* The repo already solved this twice: `changes/check.py::run_check_document` (`s19_app/tui/changes/check.py:68`) is a headless engine injected into a pre-declared service seam (`change_service.check_runner`, `change_service.py:342`), and `ChangeSummary` (`s19_app/tui/changes/model.py:373`) is the structured-result precedent with an injectable `now_fn` clock (`changes/apply.py:217`) and deterministic `to_dict()`. Option (c) is therefore not a different shape but the instruction to build (b) using these exact idioms — which is what §3/§4 encode.
  **Decision:** (b) implemented per (c)'s idioms. **Risk:** mild speculation — an ABC + registry for three no-ops is more structure than (a); accepted because the follow-up fill-in is explicitly committed in the story, and the structure is ~3 small files. **What would change it:** if the operator states the follow-up operation will NOT consume a structured result (e.g. it is purely a TUI display action), option (a) inside the existing `change_service` would suffice.
