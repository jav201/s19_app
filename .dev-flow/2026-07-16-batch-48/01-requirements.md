## 1. Introduction

### 1.1 Purpose
Define the requirements for **Batch B of the screen-upgrade "visual data-insight layer"**: taking batch-46's three-window Patch Editor from its structural layout to the **BIG** tier. This is a **render-only** batch — no parser, engine, validation, or patch-application behavior change. It surfaces data the system already computes (`CheckRunResult` aggregates + per-entry results, `LoadedFile.mem_map`, the undo/redo stack depths, the paste cap) through richer rendered surfaces, and creates the **chip-button CSS** that batch-47 correctly deferred for want of a consumer.

The headline is a **live before/after card**: on entry-row select, the analyst sees the image bytes currently at the entry's address next to the bytes the entry would write — a **read-only preview that applies nothing**.

### 1.2 Scope
**In scope (6 stories → 7 HLRs → R-TUI-075…R-TUI-081):**
- **US-P1** — EASY layer: window border titles + subtitles; entries-table colour roles; docked rows as colour-grouped chips; a variant+scope **line** (today the scope is legible only from a button's label).
- **US-P2** — check glyph (`✓`/`◐`/`✗`/`·`) **folded into the entries table's `Kind` cell as a leading span** (no new column), index-aligned to document order.
- **US-P3** — CHECKS pass/fail strip (counts + microbar) from `CheckRunResult.aggregates`.
- **US-P4** — JSON window: syntax-ish colouring of the pasted change-set + a paste-cap gauge (`N KB / 64KB`).
- **US-P5** — **live before/after card** on entry-row select (before ← `LoadedFile.mem_map`; after ← the entry's `encoded_bytes`).
- **US-P6** — history strip: position in the undo/redo stack + key hints.

**Explicitly OUT (non-goals):** any wiring or behavior change in the Patch Editor beyond read-only preview surfaces; Issues Report tiers (PARKED — still owns the P0 B1 Issues-paging no-op; parked ≠ fixed); raising the 120-col layout caps; Flow Builder (b-45+); the existing `#patch_before_after_button` → `action_before_after_report` **report-file** feature (US-061, `app.py:2158`/`:2639`) — a distinct, already-shipped feature that this batch neither replaces nor touches; any new file-system or execution surface; any new runtime dependency.

### 2.4 Constraints

1. **C-7 panel purity (design-critical, the batch's defining constraint).** `PatchEditorPanel` (`screens_directionb.py:2192`, spanning `2192-3468`; `compose` `:2644`) contains **ZERO** `self.app` references, **ZERO** `mem_map` references, and **ZERO** service imports — grep-verified over that span (PLAN.md fact 2). It is strictly presentational: `app.py` pushes data in via `refresh_*` / `set_*`; the panel posts messages out. **Every new datum the panel renders — `mem_map` for the card, check aggregates for the strip, history depths for the history strip — shall be threaded in as a method parameter (a new or extended `refresh_*` signature), following the batch-47 Inc-6 precedent `MemoryMapPanel.render_ranges(…, mem_map: Optional[Mapping[int, int]] = None)` (`screens_directionb.py:1334-1341`).** An LLR (or an implementation) that reaches into `self.app` from the panel is **wrong** and would be the panel's FIRST C-7 violation. Duck-typed `Sequence[object]` / `Mapping` parameter types are the established idiom for keeping the view widget import-free (`refresh_entries` docstring, `:3229-3232`; `refresh_check_results` docstring, `:3446-3448`).

2. **The card CANNOT use `last_summary`.** `ChangeSummaryEntry.before_bytes` is `None` for every **non-`applied`** disposition (`changes/model.py:336-338`), and a *live* card renders **BEFORE** any apply — so a live card driven by `last_summary` would show `None` before-bytes in exactly the case it exists to serve. **Before-bytes shall come from `LoadedFile.mem_map`** (`models.py:57`), read per-address at the entry's span. (PLAN.md fact 1.) This is stated as rationale on LLR-080.2 so no reviewer re-litigates it.

3. **Positional correlation is the contract; address-matching is forbidden as a join key.** Neither `ChangeEntry` (`changes/model.py:80`), `CheckRunEntry` (`:621`), nor `ChangeSummaryEntry` (`:321`) carries an id. The correlation contract is **document order** — `CheckRunEntry` records are "built by `changes.check.run_check_document`, one per document entry in document order" (`changes/model.py:660-661`), and `ChangeService.run_checks` runs the engine over **`self.document`** (`change_service.py:1258-1259`), the same document `ChangeService.rows()` renders (`:1357`). `DataTable#patch_doc_entries_table` (`screens_directionb.py:2697`) is `cursor_type="row"` → **the cursor row index IS the entry index**. **Index-align; never address-match.** An off-by-one silently mislabels a row's check result or previews the wrong entry's bytes — a wrong-answer defect with no error path — so it gets an explicit LLR (LLR-077.1) and a dedicated AT (AT-077d).

4. **C-17 — the ENTRIES TABLE is the batch's live security surface; the paste buffer is safe by construction.** ⚠ **This constraint was inverted in the Phase-1 draft and is corrected here (Phase-2 BL-1).** Two distinct surfaces, with opposite risk profiles:

   **(a) `#patch_doc_entries_table` — a LIVE, EXPLOITABLE `Text.from_markup` SINK AT HEAD, and this batch rewrites exactly those lines.** `ChangeService.check_rows`/`rows()` sets `value_text = entry.value` (`change_service.py:1402-1425`) — the **raw change-set JSON string, unmodified and file-derived** — and `refresh_entries` (`screens_directionb.py:3244-3252`) hands it to `DataTable.add_row()` as a bare `str`; Textual's `default_cell_formatter` (`_data_table.py:202-222`) sets `possible_markup=True` → **`Text.from_markup(content)`**. Verified exploitable at HEAD (probes, textual 8.2.8):

   | payload in `entry.value` | observed result |
   |---|---|
   | `[red]PWNED[/red]` | `plain='PWNED'` + `Span(0,5,'red')` — content mangled, **style injected** |
   | `[link=http://evil]click[/link]` | `Span(0,5,'link http://evil')` — **link injected from file data** (the batch-43 class) |
   | `[/nope]` | **raises `MarkupError`** → crashes `refresh_entries` |

   **This batch therefore FIXES A PRE-EXISTING LIVE VULNERABILITY**, it does not merely restyle a table. **Every cell of an entries row shall be `Text`-constructed — regardless of whether a role style is assigned to it** — and a **gate-blocking C-17 AT** over the entries table is required (LLR-075.6, AT-075e ★★). The partial-fix trap is concrete and named: a role enumeration covering only some columns leaves the rest bare `str` and the sink live.

   **(b) `CappedTextArea#patch_paste_text` (`screens_directionb.py:2906-2908`) — SAFE BY CONSTRUCTION.** Verified in the installed `textual==8.2.8`: `TextArea.get_line` (`_text_area.py:1328`) returns `Text(line_string, end="", no_wrap=True)` — the **literal constructor, never `from_markup`** — and styles resolve via `theme.syntax_styles.get(name)` (`:1501-1503`) where an **unknown token is skipped**, so payload text **cannot name a style**. The buffer holds arbitrary analyst-pasted text, but the `TextArea` render path does not markup-parse it. The C-17 requirement on it (LLR-079.3) is therefore a **preservation** requirement — do not *introduce* a markup path — not a repair of a live hole, and it remains gate-blocking (AT-079c) because the natural wrong implementation (echoing the buffer into a markup-enabled `Static`) would create one.

   **On every surface:** styling shall be applied as **styled spans over trusted-rendered literal text** — **NEVER** `Text.from_markup`, **never** an f-string of file-derived content into a markup string, **never** a `Static`/`Label` with markup enabled. The mandated hostile payload set is defined once in LLR-079.3 and reused by AT-075e. Additionally, `CheckRunEntry.linkage_symbol` and `.reason` (`changes/model.py:657-659`, file-derived) reach the NEW check-glyph and strip surfaces only as *tokens/counts*, not free text — audited in LLR-077.6 and LLR-078.5.

5. **C-29 two-axis geometry — the card is a STRUCTURAL addition.** Every "how much fits" claim in this document is flagged `assumed — pilot-measure BOTH axes in Phase 3` at 80×24 **AND** 120×30, **with the card mounted**. In particular: **the card shall NOT push the docked button rows below reachability** — that is field-audit **B2**, the exact defect batch-46 fixed, and re-litigating it is this batch's highest structural risk. AT-080d pins it explicitly, reusing batch-46's **reachable-under-scroll** contract at the 80×24 floor. Per C-29's own instruction, no acceptance threshold in this document asserts an all-at-once visible count; if a Phase-3 measurement cannot satisfy a threshold, the acceptance is relaxed **at draft-equivalent** (a §6.5 amendment), never shipped as a physically-impossible AT.

6. **Preserve every batch-46 id.** `tests/test_tui_patch_layout.py:67 _MUST_PRESERVE_IDS` is a **48-id tuple** (14 wiring-critical leaves incl. `patch_doc_entries_table` / `patch_doc_issues` / `patch_checks_results` / the 2 `Select`s + 22 census-pinned + 10 structural + 2 hidden rows). **New widgets ADD ids; no existing id may move, be renamed, or be re-parented out of its `_MUST_PRESERVE_IDS` role.** (The handoff's "14 leaf + 2 hidden" understates it — PLAN.md guardrail.)

This batch's only new CSS family is the chip-button block, whose consumer is exclusively the Patch Editor's docked rows. **The chip rules shall therefore be scoped to the patch containers (e.g. descendants of `#patch_editor_panel` `screens_directionb.py:2540`), never written as a bare `Button` / bare-class rule that a non-patch screen could match** (LLR-076.1, AT-076b). With that scoping, no cell outside the 2 patch cells drifts from the chips and C-30's sequencing mandate does not bind. **If Phase 3 finds it cannot scope the rules and must restyle `Button` app-wide, C-30 re-binds and the CSS increment shall be sequenced LAST** — that conditional is recorded here, not discovered at a gate.

### 2.5 Assumptions and dependencies
- **A1.** ⚠ **A check run depends on TWO inputs: the document AND the loaded image.** `change_service.py:1258-1259` is `result = self.check_runner(self.document, mem_map, ranges, mac_records, a2l_tags)`. Two consequences, and the Phase-1 draft stated only the first:
  - **(i) Index-alignment (the document arm).** The engine runs over **`self.document`** — the same object `rows()` renders (`:1357`) — so `last_check_result.entries[i]` corresponds to `document.entries[i]` at run time (`:1261` `self.last_check_result = result`; `changes/model.py:660-661`). *If false → HLR-077's index-alignment is invalid and the glyph must be dropped.*
  - **(ii) ⚠ Result VALIDITY (the image arm) — this is what a document-only fingerprint misses.** `CheckRunEntry.actual_bytes` is read **from `mem_map`**, i.e. from the loaded image, **not** from the document. And `ChangeService()` is constructed **once** at `app.py:1171` and is **never rebuilt on file load**; the only `last_check_result` resets are `undo` `:474` / `redo` `:506` (A2). ⇒ **Run checks against image A (all `✓`) → load image B → the document is unchanged → a document-only signature still matches → the glyphs render, describing image A.** This is reachable via the most routine action in the app (loading a file) and is the exact wrong-answer class LLR-077.2 exists to prevent. *This is why LLR-077.2's stamp is `(document_signature, image_generation)` and not a document signature alone (Phase-2 BL-4).*
  - **Scope note (so Phase 3 does not over-build):** `mac_records` / `a2l_tags` are also `check_runner` inputs but drive `CheckRunEntry.linkage` — **not** `.result` — and the glyph renders `.result` only. They therefore need **no** covering in the stamp.
- **A2.** ⚠ **`last_check_result` is NOT reset by document mutation.** Grep-verified: it is written at `change_service.py:1261` and reset **only** at `undo` `:474` and `redo` `:506` (batch-40). `add_entry` `:588`, `remove_entry` `:671`, `load` `:827`, and `load_text` `:882` leave it standing. **Today this is harmless** — `check_rows()` (`:1291`) renders self-describing rows, each naming its own address range (`:1345-1349`), and makes no alignment claim. **The glyph column CREATES the alignment claim**, so this batch must add the provenance guard (LLR-077.2). *This assumption is the reason HLR-077 has a provenance LLR at all; if a future batch resets `last_check_result` on every mutation, LLR-077.2 becomes redundant but not wrong.*
- **A3.** `CheckRunResult.aggregates` always carries all three keys, even at zero (`CHECK_AGGREGATE_KEYS` `changes/model.py:565-571`, "all three always present, even when zero … so report tables never branch on missing keys"). *If false → HLR-078's strip must branch on missing keys.*
- **A4.** `LoadedFile.mem_map` is a `Dict[int, int]` sparse map (`models.py:57`); an address absent from it is **unmapped**, not zero. The card renders unmapped before-bytes as a distinct placeholder, never as `00` (LLR-080.3, AT-080c).
- **A5.** The entries table applies **no row-level style override** — `refresh_entries` (`screens_directionb.py:3244-3252`) calls `table.add_row(row.kind_text, …)` with plain values and no `.style`. Therefore per-cell colour roles (HLR-075) **will be visible on the live table** — unlike batch-47's A2L table, where a row-level severity override subsumed the accents (batch-47 §6.5 Amendment E). *This assumption is checked explicitly because the sibling batch shipped an amendment for exactly its inverse; if Phase 3 finds an override, HLR-075's accent clause is scoped builder-only via a §6.5 amendment, following Amendment E's precedent.*
- **A6.** `insight_style` is available and non-frozen with the helpers this batch consumes: `human_bytes` `insight_style.py:68`, `label_value` `:119`, `microbar(frac, width, style="", floor=False)` `:160`, `threshold_style(pct, warn, bad)` `:224`; palette `PURPLE` `:50`, `CYAN` `:52`, `VALUE` `:36`, `GREEN` `:38`, `YELLOW` `:40`, `RED` `:42`, `DGRAY` `:48`, `HILITE` `:46`, `LBLUE` `:47`. **`microbar(floor=True)` is ONLY for bars meaning "this exists"; proportional bars stay unfloored** (batch-47 rule) → HLR-078's pass/fail bar is **`floor=False`**.
- **A7.** **Chip-button CSS does not exist.** The only chip-like rule today is `.issue-code-chip` (`styles.tcss:1023`), which styles a `Label`, not a `Button`. Batch-47's deferral was therefore correct; **this batch creates the family**, and the patch docked rows are its first and only consumer. Patch CSS block ≈ `styles.tcss:824-1170`; `.patch-window-body` `:893`; `.patch-docked-row, .patch-docked-group` `:902-903`.
- **A8.** ⚠ **Tree-sitter is NOT installed.** Probe executed 2026-07-16 in the batch-48 worktree: `import tree_sitter` → `ModuleNotFoundError`; `textual._tree_sitter.TREE_SITTER` → `False`; `TextArea('{"a": 1}', language='json')` **constructs without raising and silently applies no highlighting**. `TextArea.available_languages` listing `'json'` is therefore a **false affordance** — the name is registered, the grammar is not. *Consequence: `language="json"` is a no-op, and HLR-079's colouring needs the LLR-079.1 mechanism (or its fallback rung), not a one-line `language=` change. This probe is the single most load-bearing draft-time finding for US-P4.*
- **A9.** `TextArea` exposes an instance `_highlights` (probe 2026-07-16: `hasattr(ta, "_highlights") → True`, type `collections.defaultdict`) and `TextAreaTheme` exposes `syntax_styles` / `get_highlight` / `register_theme` (probe: present on `textual.widgets.text_area.TextAreaTheme`). These are **private/semi-private Textual internals** → `assumed — Phase-3 verify against the textual==8.2.8 pin`, with the LLR-079.1 fallback rung pre-committed and a feature-detect guard mandated.
- **A10.** The autonomous-through-self-merge run mode is granted **for this batch only** (kickoff, per `feedback_standing_auth_per_batch`), gated on a clean final independent PR-level QA.

**LLR-077.2 — TWO-PART provenance stamp: glyphs only for a result current to the live document AND the live image ★**
- **Traceability:** HLR-077
- **Statement:** `ChangeService` shall record, at check-run time, a **two-part stamp** `(document_signature, image_generation)` describing the inputs the run was executed over, and `rows()` shall emit result glyphs **only when BOTH parts of the live stamp equal the recorded stamp**; on any mismatch every row's glyph shall be the no-result `·`.
  - **`document_signature`** shall be derived from the ordered per-entry tuple `(entry_type, address, encoded_bytes)` so that an added, removed, reordered, or **in-place-edited** entry all invalidate it.
  - **`image_generation`** shall be a **monotonic token owned by `app.py`**, bumped in `_apply_loaded_file` on every image load and pushed into `ChangeService` alongside `mem_map`, so that loading a different image invalidates the stamp even when the document is untouched.
- **Why the image arm exists (Phase-2 BL-4 — the draft shipped a document-only stamp and it was insufficient):** `check_runner(self.document, mem_map, ranges, …)` (`change_service.py:1258-1259`) reads each entry's `actual_bytes` **from the image**, and `ChangeService()` is constructed **once** at `app.py:1171` and **never rebuilt on load** (A1-ii). ⇒ **check image A (all `✓`) → load image B → document unchanged → a document-only signature MATCHES → the glyphs render, describing image A.** Reachable via the most routine action in the app, with no error path. The document arm alone does not see it.
- **Mechanism (normative) — cheapest exact answer:**

2. **The card reads `mem_map`, not `last_summary`.** `ChangeSummaryEntry.before_bytes` is `None` for every non-`applied` disposition (`changes/model.py:336-338`) and the card renders **pre-apply** — `last_summary` is structurally incapable of serving it (§2.4-2).
3. **The glyph rides on `ChangeEntryRow`; `mem_map` cannot.** The glyph is derivable inside `rows()` (which already owns `last_check_result`), so a defaulted `check_glyph` field reaches all three `refresh_entries` call sites free (LLR-077.5). The card needs `mem_map` at **selection** time, so it must be retained on the panel → a signature change at all three sites (LLR-080.2, §6.4).
4. **A TWO-PART provenance stamp, not a mutation reset (LLR-077.2).** Resetting `last_check_result` on `add_entry`/`remove_entry`/`load` would clear the CHECKS panel — a user-visible behavior change, out of scope for a render-only batch. The stamp gives correctness with zero behavior change. **Count-equality was considered and rejected** (the batch-38 in-place per-entry JSON edit preserves the count while changing the entry), **and a document-only signature was shipped in the draft and rejected at Phase-2 review** — the run's `actual_bytes` come from the **image**, and the service is never rebuilt on load, so check-A → load-B → glyphs-describe-A passes a document signature (BL-4). Hence `(document_signature, image_generation)`, with `image_generation` an `app.py`-owned monotonic token bumped in `_apply_loaded_file` (and **not** `id(mem_map)` — id reuse after GC would re-create the bug). **The reset alternative was re-examined at Phase-2 and is NOT simpler and NOT more correct:** it is a user-visible behaviour change in a render-only batch **and** would still need the image-load trigger — i.e. BL-4's work anyway. **Verdict: stamp, widened.**
5. **JSON colouring: IN-PLACE `_highlights`, decided by the operator (LLR-079.1).** `language="json"` is a **silent no-op** without tree-sitter (probe A8), and `textual[syntax]` is struck as a dependency/lock-in change in a render-only batch. The Phase-1 docs were **unreconciled** (this doc: in-place; `01b`: a preview widget) — **operator decision (AskUserQuestion 2026-07-16): IN-PLACE**, because it is the only route that fits the ~5-content-row 80×24 floor, the path is **verified** C-17-safe by construction, and the failure mode is cosmetic-only behind a feature-detect. **The C-13.1 "ladder" framing is dropped** (MJ-8): gauge-only recovers **zero** of the colouring deficit, so it is an acceptance relaxation (§6.5 Amendment B), not a mechanism rung. **Stated plainly: in-place, else Amendment B.** The fallback path is itself tested (AT-079d) because CI pinned at 8.2.8 can never exercise degradation naturally. Markup-rendering the paste is struck as a C-17 violation either way.
9. **The glyph folds into the `Kind` cell; no column is added (LLR-077.4, BL-3).** The house idiom, verified in source: A2L `app.py:9548` (*"the name cell (index 0) carries the leading in-image glyph"*) and MAC `app.py:9223-9226` (*"**Fold** a leading status glyph … into the Tag cell **as its own span**"*) — batch-47 shipped exactly this as LLR-068.1/070.1 **without amending the table's requirement**. Rejected: a **leading column**, which would shift `Coordinate(row,1)`/`(row,2)` under the existing index-readers (`tests/test_tui_patch_editor_v2.py:2578`, `:3208-3209` — qa's T-1, *"the batch's sharpest regression risk"*), and would manufacture a 6-columns-at-80 width deficit that the draft then **pre-committed Amendment C to pay for** — the tell that the choice was wrong. Under the fold, T-1 **dissolves** (col 0 is unasserted → 0 edits to that 32-hit census file) and **Amendment C is retired**. Not redundant with `Status`: `status_text` is the **containment** verdict, the glyph the **check-run** verdict — different semantics, different lifetimes.
10. **The entries table's C-17 fix is a first-class, gate-blocking requirement (LLR-075.6, BL-1).** The draft's threat model was **inverted**: it named the paste buffer "the batch's only untrusted surface" (it is untrusted but **safe by construction** — `TextArea.get_line` uses the literal `Text` constructor and unknown style names are skipped) while treating the **entries table** — a **live, proven-exploitable `Text.from_markup` sink at HEAD** — as a styling change. Rejected: leaving the safety clause folded into LLR-075.2's role enumeration, which enumerated **3** roles for **5** columns and thereby licensed the partial fix (`status_text` + `linkage_text` left bare `str`, sink still live). The construction requirement is now separate from the styling requirement, covers all five cells regardless of role, and carries its own gate-blocking AT (AT-075e). **This batch fixes a pre-existing live vulnerability.**
6. **Chip CSS is patch-scoped → C-30 = N/A, with a falsifiable probe (LLR-076.1, AT-076b).** The verdict rests entirely on the scoping; the leak probe is what turns it from an assertion into evidence. The re-bind condition is written into §2.4-8, not left for a gate.
7. **The gauge reports the unit the cap enforces (chars).** `_CLIPBOARD_READ_CAP_CHARS = 65536` is measured in **chars** (`os_clipboard_input.py:72`); for non-ASCII pastes chars ≠ bytes, so a byte-denominated gauge would disagree with the truncation the user experiences (LLR-079.4). The `KB` vs `KiB` label spelling (batch-47 Amendment D made `human_bytes` binary) is a Phase-3 vocabulary decision; the numeric contract is what AT-079a asserts.
8. **The strip's bar is unfloored.** `microbar(floor=True)` is reserved for "this exists" bars (batch-47); a pass-rate bar is proportional (LLR-078.4).
