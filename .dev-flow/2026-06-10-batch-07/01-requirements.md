> **Strict normative convention — IEEE 830 + EARS**
> - `shall` = normative, binding, verifiable requirement. **Only** inside HLR / LLR statements.
> - `should` = informative / explanatory text, **NOT binding**. **Only** outside HLR / LLR statements.
> - Any `should` inside an HLR / LLR statement is a **writing error** → phase-2 blocker.
> - `may` = optional. `will` = future declaration or fact about an external actor.

> **Verifiability rule** — every `test`/`analysis` requirement carries an **Executed verification** (exact command) AND a **Numeric pass threshold**. Missing either is a phase-2 blocker. `demo`: observable procedure + named qualitative criterion. `inspection`: file/commit/section + observable condition.

> **LLR symbol-citation rule** — any named code symbol/widget-id/constant cites a grep-verified `file:line`, or is flagged `NEW — created in Phase 3`; layout/magic-number constants cite a measured value + method or are flagged `assumed — verify in Phase 3`. Two phase-2 blockers enforce this.

> **Environmental-measurement citation rule — extends the LLR symbol-citation rule.** Any constant describing the runtime or layout **environment** — container/parent widths, derived geometry (e.g. `body_w`, pane shares), responsive breakpoints and transition points, timing/latency budgets, platform or CI environment values — MUST cite, at draft time: **(a) WHERE it was measured** (the probe or test `file:line`, or the exact `App.run_test(size=...)` / command invocation), **AND (b) the REGIME/CONDITIONS under which the measurement holds** (terminal-size band, CSS class state, rail/panel visibility, platform, dataset size). A measurement applied **outside its measured regime** MUST be re-measured in that regime or flagged `assumed — verify per-regime`. **Derived numbers inherit the flag.** **Phase-2 blocker classes:** (a) an environmental constant asserted as fact whose citation lacks its measurement conditions; (b) a constant or its derivatives applied in a regime other than the one cited. ## 2. Overall description

### 2.1 Product perspective
Batch-03/04 built a three-subflow change system: parameter changes by A2L name (CDFX XML), raw memory changes by address, and a unified JSON container with selective export — surfaced in one Patch Editor screen with ~14 widgets and 11 routed actions. The operator wants **one** declarative JSON system, hex-first, plus three capabilities that do not exist today: declarative expected-value checks, a project report, and multi-S19 variant projects (today hard-limited to 1 S19 + 1 MAC + 1 A2L).

### 2.2 Product functions
1. Declare, validate, and apply hex-first JSON change files (string patches + single-address patches, encoding-aware, collision-checked) with a summary annotating MAC/A2L linkage.
2. Declare and execute check files (expected values vs loaded memory) per project, automatable.
3. Generate a Markdown project report: modified files, before→after values, executed checklists, ±64-byte (adjustable) hexdumps of modified regions.
4. Hold N S19 variants per project sharing one A2L + one MAC; apply change/check files in batch or per-variant; reflect everything in the report.

### 2.3 User characteristics
Operator (firmware/test engineer) and QA-responsible operator producing auditable evidence per project.

### 2.4 Constraints
- Engine/parser layers consumed read-only; new logic lives in services + new modules per CLAUDE.md (app.py stays orchestration-only).
- ≤5 files per increment; supervised gates per increment. One explicit, operator-approved exception: increment E3b (retirement enactment) carries a ~20–25-file budget, predominantly deletions and test-file edits (§6.5; operator decision 2026-06-10, D-2).
- CI two-tier gate (PRs `-m "not slow"`, full on `main` push) lands in this batch's first PR.
- No SVG snapshot baselines regenerated locally (project memory).
- Workspace layout rules (`.s19tool/` structure, `sanitize_project_name`, `resolve_input_path`) are the base for the variants + reports extensions.

### 2.5 Assumptions and dependencies
- The batch-03/04 `cdfx/` modules are the starting point: `memory.py`/`memory_validate.py`/`changeset.py`/`unified_io.py` evolve into the v2 system; `changelist.py` (parameter-by-name) and the CDFX XML I/O **retire** (operator decision: address-only entries).
- The cross-artifact `range_index` + validation engine provide the MAC/A2L linkage lookups (read-only).
- `LoadedFile` single-file snapshot model must grow a variant dimension (US-005) — the largest structural change; Phase 1 must design it explicitly.
- Report hexdump rendering reuses `hexview.py` primitives where possible (`render_hex_view_text` or a non-TUI sibling).

## 3. High-level requirements (HLR)

> Mapping: US-002 → HLR-001/002/003 · US-003 → HLR-004 · US-005 → HLR-005/006 · US-004 → HLR-007/008. Drafted by two parallel architects (A: 001–004, B: 005–008), interfaces reconciled in §6.2 C-6. All anchors re-grepped 2026-06-10.

### HLR-001 — Hex-first v2 change-file format and reader/validator (US-002, E1)
- **Statements (EARS):** (1) The system shall define a single JSON change-file format (v2) whose document declares `format`, `version`, `kind`, `encoding`, and `value_mode` metadata and a single `entries` array containing only address-keyed entries of two kinds: **string patch** and **byte patch**. (2) The v2 reader shall accept no symbolic addressing field in any entry; every entry shall be located exclusively by an explicit hex address. (3) When a v2 change file is read, the reader shall collect every schema, metadata, per-entry, collision, and resource-ceiling finding as a `ValidationIssue` without raising and shall always return a document object (possibly empty). (4) If two entries in one document declare intersecting target byte ranges, the reader shall record an ERROR-severity collision finding and the document shall be marked not applicable.
- **Traceability:** US-002; §6.2-1 (address-only); supersedes batch-04 HLR-005/006/008.
- **Validation:** test · **Executed verification:** `pytest -q tests/test_changes_schema.py tests/test_changes_collision.py` (NEW) · **Threshold:** 100% pass; collision suite ≥6 parametrized overlap geometries with exact expected issue counts. · **Priority:** Must.

### HLR-002 — Apply engine, change summary, linkage annotation (US-002, E2)
- **Statements (EARS):** (1) When the operator triggers Apply on a validated change document, the system shall write each applicable entry's encoded bytes into the loaded image's memory map and shall capture the prior byte values of every written range before mutation. (2) If a change document carries any ERROR-severity issue, the apply engine shall perform no memory writes. (3) The apply engine shall produce a change-summary object recording, per entry: target address range, before bytes, after bytes, disposition (applied / skipped-partial / skipped-outside / skipped-no-image / blocked), and an informative linkage classification (standalone / MAC-linked / A2L-linked / both). (4) The linkage classification shall be computed via the sorted-range primitives against the loaded MAC records and A2L tag ranges and shall not influence whether an entry is applied. (5) When an apply completes with ≥1 applied entry on an S19-loaded image, the system shall offer to persist the patched image to the project work area under an **operator-provided filename** (operator decision 2026-06-10); when the loaded image is Intel HEX, the system shall state that HEX save-back is not supported this batch and shall persist nothing (S19-only save-back — operator decision 2026-06-10, D-1).
- **Traceability:** US-002 (summary + linkage); §6.2-1 (linkage informative only).
- **Validation:** test · **Executed verification:** `pytest -q tests/test_changes_apply.py tests/test_changes_linkage.py tests/test_tui_patch_editor_v2.py::test_save_back_prompt tests/test_tui_patch_editor_v2.py::test_declaration_faults_visible` (NEW) · **Threshold:** 100% pass; exact before/after byte tuples for ≥4 dispositions; all 5 dispositions (incl. `blocked`, via LLR-002.1) asserted; all 4 linkage classifications asserted. · **Priority:** Must.

### HLR-003 — Single JSON-flow Patch Editor; cfdx/.cdfx retirement (US-002, E3a/E3b)
- **Statements (EARS):** (1) The Patch Editor shall present exactly one change-flow section operating on v2 JSON documents (entry table, entry add/edit/remove, load/validate/apply/save controls). (2) The system shall provide no CDFX XML read, CDFX XML write, parameter-by-name change list, or selective `.cdfx` export capability. (3) If the operator loads a `.cdfx` file or a v1 unified JSON document, the system shall report one ERROR-severity unsupported-format finding naming the v2 format, and shall not crash.
- **Traceability:** US-002 locked decision (cfdx retired); REQUIREMENTS.md §8–9 supersession (Phase 6).
- **Validation:** test + inspection · **Executed verification:** `pytest -q tests/test_tui_patch_editor_v2.py` (NEW) + inspection `git ls-files s19_app/tui/cdfx/` empty after E3b · **Threshold:** 100% pass; lean suite green after E3b with the §6.6 measured disposition table fully enacted (every RETIRE/REWRITE row executed; SURVIVES rows still passing). · **Priority:** Must.

### HLR-004 — Declarative check files, headless-runnable (US-003, E4)
- **Statements (EARS):** (1) The system shall accept check documents in the same v2 JSON schema family, discriminated by the top-level field `kind = "check"`, whose entries declare **expected** values at explicit hex addresses. (2) When a check document is executed against a loaded image, the system shall produce, per entry, exactly one result of pass, fail, or uncheckable, where uncheckable covers any entry whose target range is not fully inside the loaded image (or no image is loaded). (3) The system shall expose a service-level entry point that executes a check document against project files and returns the results object without any TUI interaction. (4) The check-results object shall carry, per entry, the target address range, expected bytes, actual bytes (when readable), the result, and the informative linkage classification, plus aggregate pass/fail/uncheckable counts and the check document's collected declaration-fault issues (the carrier that brings check declaration faults to the report — B-2).
- **Traceability:** US-003.
- **Validation:** test · **Executed verification:** `pytest -q tests/test_checks_engine.py` (NEW; imports no Textual app modules) · **Threshold:** 100% pass; exact result triples on a fixture with ≥1 of each result kind; no-Textual-import verified via a subprocess-isolated probe or static import-graph inspection (in-session `sys.modules` assertions are unreliable — F-Q-07, LLR-004.4). · **Priority:** Must.

### HLR-005 — Multi-variant project model (US-005, E5a/E5b)
- **Statement (EARS):** When a project directory under `.s19tool/workarea/<project>/` contains N ≥ 1 S19/HEX files, at most one MAC file, and at most one A2L file, the system shall accept the project as valid, shall enumerate the S19/HEX files as an ordered variant set with a deterministic order, and shall render exactly one active variant in the TUI at any time, parsed on a worker thread and applied on the main UI thread.
- **Traceability:** US-005; §6.3 LoadedFile risk (mitigated by §6.2 C-7 design).
- **Validation:** test · **Executed verification:** `pytest -q tests/test_workspace_variants.py tests/test_tui_variants.py -m "not slow"` (NEW) · **Threshold:** 100% pass incl. backward-compat case (single-S19 project loads identically). · **Priority:** High (blocks E6, E7).

### HLR-006 — Batch / per-variant execution (US-005, E6)
- **Statement (EARS):** When the operator or a headless caller requests execution of one or more change files and/or check files against a multi-variant project, the system shall execute them per the project's assignment model — batch (every assigned file against every variant) or an explicit variant→file mapping — in deterministic order, shall isolate per-variant failures so that one variant's error never aborts the remaining variants (collect-don't-abort), and shall produce exactly one change-summary object and/or one check-results object per assigned variant.
- **Traceability:** US-005; US-003 (automation chain); §6.5 dependency E6→E4+E5b.
- **Validation:** test · **Executed verification:** `pytest -q tests/test_variant_execution.py -m "not slow"` (NEW; includes injected-failure case) · **Threshold:** 100% pass; injected-failure case: results count == variant count. · **Priority:** High (blocks E7).

### HLR-007 — Markdown project report generation (US-004, E7)
- **Statement (EARS):** When report generation is requested for a project, the system shall write a Markdown document to `.s19tool/workarea/<project>/reports/<timestamp>-report.md` containing: (a) the project and variant inventory, (b) the modified files, (c) per-modification before→after values with linkage annotation, (d) each executed checklist with per-entry pass/fail/uncheckable results, (e) a hexdump of every modified region with ±`context_bytes` (default 64, adjustable per invocation) of surrounding memory, and (f) one section per variant plus a consolidated overview; and where size caps are exceeded the system shall truncate with an explicit in-document marker stating the omitted count.
- **Traceability:** US-004; §6.2-3 (Markdown + path locked).
- **Rationale (informative — operator, 2026-06-10 gate):** the linkage column exists so the before→after report lets the operator judge whether a change *makes sense* — e.g. spotting that a write landed on an A2L multiplier instead of a quantity. This sense-check is deliberately OBSERVED in the report, never BLOCKED by rules: changes are address-only against the hex/S19, and no A2L/MAC change rules exist because only the image is modified. The report is where that human validation happens.
- **Validation:** test · **Executed verification:** `pytest -q tests/test_report_service.py -m "not slow"` (NEW) · **Threshold:** 100% pass; all six section classes (a)–(f) present; hexdump window spans exactly `region ± context_bytes` rounded to 16-byte rows. · **Priority:** High.

### HLR-008 — Report TUI view + headless generation entry (US-004, E8)
- **Statement (EARS):** When the operator invokes the report viewer in the TUI, the system shall list the active project's reports newest-first and shall render the selected report read-only; and the system shall expose a headless service entry point that generates a report without any TUI interaction.
- **Traceability:** US-004 ("also viewable in the TUI"); US-003 (headless chain).
- **Validation:** test + demo · **Executed verification:** `pytest -q tests/test_report_service.py::test_generate_report_headless_no_app tests/test_tui_report_view.py -m "not slow"` (NEW) · **Threshold:** 100% pass; headless test constructs no Textual `App`. **Demo criterion:** operator opens the viewer on a 2-variant fixture project and scrolls a rendered report; report body readable, no raw-markdown leakage in headings. · **Priority:** Medium (E8).

## 4. Low-level requirements (LLR)

> Full LLR statements with embedded Executed verification + Numeric pass threshold. Every named symbol carries a re-grepped `file:line` (2026-06-10) or a `NEW — created in Phase 3` flag; size constants carry measurement+conditions or `assumed — verify per-regime`.

### HLR-001 LLRs — format + reader/validator

- **LLR-001.1 — v2 document metadata.** The v2 document shall be a single JSON object carrying `format` = `"s19app-changeset"` (NEW, replacing `UNIFIED_FORMAT_ID`, `unified_io.py:99`), `version` = `"2.0"` (NEW, replacing `unified_io.py:105`), `kind` ∈ {`"change"`, `"check"`}, `encoding` (a Python codec name, e.g. `"utf-8"`), `value_mode` ∈ {`"text"`, `"codes"`}, and `entries`. *Verify:* `tests/test_changes_schema.py::test_metadata_roundtrip` — write→read preserves all 5 metadata fields; 100% of 5 assertions.
- **LLR-001.2 — entry kinds and shapes.** Each entry shall carry `type` ∈ {`"string"`, `"bytes"`} and `address` (JSON string matching `^0x[0-9A-Fa-f]+$` or non-negative integer; canonical writer emits the `0x` string form). A `"string"` entry shall carry `value`; a `"bytes"` entry shall carry `bytes` as whitespace-separated two-hex-digit tokens — this strict token grammar is the **wire grammar** of the file format, and the reader shall reject any other `bytes` spelling. The permissive grammar of `parse_new_bytes` (`services/cdfx_service.py:312` — also accepts commas, decimals, `0x` prefixes; `parse_address` `:269`) is the **TUI-input grammar only**: interactive entry fields may accept it and shall normalize to the canonical wire form on document write; it never defines what is valid in a file (grammar split per §6.2 C-1 — preserves the 4 parse-helper SURVIVES rows, F-Q-04). In `value_mode="text"`, target bytes are `value.encode(encoding)`; in `value_mode="codes"`, `value` is an array of code points joined and encoded with `encoding`. *(Deliberate departure from batch-04 DD-10 integer-only address — `0x` string field is unambiguous and hex-first; gate-confirmable.)* *Verify:* `::test_entry_shapes` ≥8 accept/reject cases; 100%.
- **LLR-001.3 — metadata validation, collect-don't-abort.** Unrecognized `format`/`kind`/`value_mode` or an `encoding` that does not resolve to a **text** codec shall each record one ERROR issue (NEW codes `CHG-FORMAT`, `CHG-KIND-UNKNOWN`, `CHG-VALUE-MODE-UNKNOWN`, `CHG-ENCODING-UNKNOWN`). Encoding acceptance shall require `codecs.lookup(encoding)._is_text_encoding` to be truthy — `codecs.lookup` succeeding alone is NOT sufficient (verified live in review F-S-02: `codecs.lookup("zlib_codec")` succeeds, then `str.encode` raises `LookupError`, escaping the `UnicodeEncodeError` handler), so non-text codecs (e.g. `zlib_codec`, `base64_codec`) are rejected at metadata level. Structural read codes `MF_JSON_PARSE`/`MF_BAD_STRUCTURE`/`MF_SIZE_CAP`/`MF_PATH_UNRESOLVED`/`MF_ENTRY_LIMIT` (`unified_io.py:122-140`) shall be reused with spellings preserved (constants migrate). When any metadata-level ERROR is recorded, the returned document shall carry **zero entries** — entry content is not interpreted under a faulted envelope (F-A-16). *Verify:* `::test_metadata_faults` 7 crafted-fault cases (the 6 prior + `encoding="zlib_codec"`), one exact code each, plus a zero-entries assertion per metadata-fault case; 100%.
- **LLR-001.4 — per-entry validation.** Malformed `address`/`bytes`, empty `value`/`bytes`, or an encode failure under the declared encoding shall each record one ERROR issue (`CHG-ADDRESS-SYNTAX`, `CHG-BYTES-SYNTAX`, `CHG-VALUE-EMPTY`, `CHG-ENCODE-FAIL` — NEW) naming the entry's address/index — never raw byte content (C-9 confidentiality, `memory_validate.py:33-36`) — and skip the entry while continuing. Encode-failure coverage shall span `UnicodeEncodeError`, `LookupError`, `ValueError`, and `OverflowError` (verified live in review F-S-02: `value_mode="codes"` with a code point > `0x10FFFF` raises `ValueError`, not `UnicodeEncodeError`); no encode path shall raise out of the reader. *Verify:* `::test_entry_faults` ≥8 cases incl. latin-1-only char under `ascii`, code point `0x110000` under `value_mode="codes"`, and a negative code point; 100% + no-raw-bytes assertion.
- **LLR-001.5 — target-range derivation and intra-file collision.** Target range = `(address, address + encoded_byte_length)` (string: `len(value.encode(encoding))` after `value_mode` resolution; bytes: token count) — the `MemoryChange.addressed_range` semantics (`cdfx/memory.py:146-158`) extended to encoded strings. Intersecting ranges (`a0 < b1 and b0 < a1`, predicate of `_overlap_issues` `memory_validate.py:175-230`) or identical addresses shall record one ERROR `CHG-COLLISION` (NEW) per colliding entry naming both addresses. *(Severity promotion: batch-04 `MEMORY_OVERLAP_CODE` WARNING → ERROR; identical-address silent dedup (`memory.py:228-280`) → explicit error. Both gate-confirmable; resolves disposition questions D1/D2 in §6.6.)* *Verify:* `tests/test_changes_collision.py` — 6 geometries with exact issue counts; 100%.
- **LLR-001.6 — image-containment status.** With an image loaded, each entry shall be stamped `MemoryStatus` (`cdfx/memory.py:38-64`) against `LoadedFile.ranges` (`models.py:39`) via `build_sorted_range_index`/`range_in_sorted_ranges` (`range_index.py:9,71`) — `_range_status` (`memory_validate.py:72-116`) reused; with no image, every entry is `UNVALIDATED_NO_IMAGE` with no issues. *Verify:* `tests/test_changes_containment.py` 5 cases; 100%.
- **LLR-001.7 — resource ceilings and path resolution.** The reader shall resolve via `resolve_input_path` (`workspace.py:357`), enforce `READ_SIZE_CAP_BYTES` (`unified_io.py:157`) pre-parse, `MF_ENTRY_COUNT_CEILING` (`:168`) on entries, `MF_RUN_LENGTH_CEILING` (`:175`) on **encoded** byte length, with a **pre-encode guard**: the raw `value` character count / `codes` array length shall be checked against the same ceiling BEFORE encoding, so an oversized declaration can never force an unbounded encode (F-S-04); canonical writes use the staged-containment pattern of `write_unified_to_workarea` (`unified_io.py:348-461`). *Verify:* `::test_resource_ceilings` 5 cases incl. one pre-encode rejection (oversized `value` never reaches `.encode`); 100%.
- **LLR-001.8 — v1 hard break.** A document with `format` `"s19app-unified-changeset"` (`unified_io.py:99`) or matching `_is_unified_shape` (`unified_io.py:752-782`) shall record exactly one ERROR `CHG-V1-FORMAT` (NEW) naming the v2 token, return an empty document; v1 detection shall run BEFORE generic format validation and, on match, shall suppress the generic `CHG-FORMAT` finding, so a v1 file yields exactly one format finding — `CHG-V1-FORMAT`, never both (precedence rule resolving the 001.3/001.8 contradiction — F-A-03); no read-compat shim shall exist. *(Recommended §6.2 C-3; gate-confirmable.)* *Verify:* `::test_v1_rejected` — real batch-04 fixture → exactly 1 issue, 0 entries; 100%.

### HLR-002 LLRs — apply engine + summary + linkage

- **LLR-002.1 — apply gate.** Any ERROR issue, or `kind` ≠ `"change"` → the apply engine (`apply_change_document`, NEW) shall perform zero writes and return a summary with every disposition `blocked`. *Verify:* `tests/test_changes_apply.py::test_error_blocks_apply` — `mem_map` dict-equal before/after; 0 mutated keys.

**Phase-1 design decisions (architect A — US-002/003):**
- **C-1 — v2 schema sketch:**
```json
{
  "format": "s19app-changeset",
  "version": "2.0",
  "kind": "change",
  "encoding": "utf-8",
  "value_mode": "text",
  "entries": [
    { "type": "string", "address": "0x80001000", "value": "687000123" },
    { "type": "bytes",  "address": "0x800020",   "bytes": "FF" },
    { "type": "bytes",  "address": "0x80002F00", "bytes": "DE AD BE EF" }
  ]
}
```
A check file is identical with `"kind": "check"` (entries read as expectations). `value_mode: "codes"`: `"value": [54, 56, 55]` — code points encoded via `encoding`. **Grammar split (F-Q-04):** the `bytes` **wire grammar** is strict — whitespace-separated two-hex-digit tokens only, nothing else valid in a file; the permissive `parse_new_bytes` grammar (commas, decimals, `0x` prefixes) is a **TUI-input convenience only** that normalizes to the wire form on document write.
