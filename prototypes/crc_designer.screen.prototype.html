CRC Algorithm Designer — TUI design

Every field, verdict, and JSON block below binds to a real, tested type from the built keel: the form maps to crc_kernel.CrcAlgorithm, the coverage strip to crc_designer_model.CrcTarget, the JSON to emit_template / parse_job, and the hero verdict to CrcAlgorithm.kat_ok() (a genuine tri-state). All hex values are the actual engine outputs.

Two view-boundary variants: Variant A = authoring bench only; Variant B (chosen) adds multi-range coverage preview over the loaded image.

Variant A — Authoring bench only

Build + KAT-verify + save/load an algorithm template (*.crc.json, pure math). Running over firmware stays in the CRC operation / CRC block.

JSON preview = emit_template() output. The template has NO addresses — it is reusable.

Variant B — Bench + multi-range coverage preview (chosen)

Everything in A, plus a per-firmware job: an algorithm_ref + one or more CrcTargets (multi-range, with the intra×join gap toggles), previewed over real bytes.

JSON preview = a parse_job()-shaped job. Preview only — never writes firmware.

How it binds to the built types

  • Form → crc_kernel.CrcAlgorithm: width / poly / init / refin / refout / xorout / check are its exact fields. The preset selector is PRESETS; picking one populates the form (preset_by_name).
  • Hero verdict → CrcAlgorithm.kat_ok(): a real tri-state — True = MATCH, False = MISMATCH, no check = no-expected (when check is unset). The view renders the returned state directly; no extra logic.
  • Coverage strip → crc_designer_model.CrcTarget: its fields, 1:1 — ranges[] (add/remove), intra_gap, join, pad_byte, on_gap_conflict, output_address, store_width, store_endianness. The two CRCs are compute_target_crc; the stored bytes are store_word.
  • Gap safety → on_gap_conflict (obs #2): before filling, the engine checks the gap against the real image; on stray data it aborts (default) rather than emit a CRC that silently diverges from the device. Backed by gap_conflict() in the runnable prototype.
  • JSON blocks are the serializers: Variant A = emit_template() (template, no addresses); Variant B = a parse_job()-shaped job (algorithm_ref + targets[]). Load routes through read_template / parse_job, collect-don't-abort — a fault is one string in the warn row.
Honest numbers, from the tested engine. 0xCBF43926 (seed = zlib), the preset table, the mismatch 0x8D94, the coverage oracles concat=0x9C5BCBBD / fill=0x2A8A3950, and the little-endian store 50 39 8A 2A are all real outputs asserted in tests/test_crc_kernel.py / tests/test_crc_designer_model.py (28 passing); the gap-safety abort is backed by gap_conflict() in the prototype. Production refinements: check is always the CRC of "123456789" (standard KAT, validated on save — #3); performance (256-entry LUT built at init) and shift-direction (reflected → LSB-first) are engine concerns, not template fields (#1, #4). The one piece the JSON implies but the keel doesn't yet emit is a whole-job serializer (emit_job) — the flagged follow-up.
